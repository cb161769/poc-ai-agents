Pipeline de run_poc_loop.sh

sin argumentos — el escenario lo decide el ticket real de Jira. Orquestación a demanda, no CI/CD: nada dispara esto automáticamente cuando aparece un commit nuevo — ver nota en "Estructura del repositorio" más abajo.

01
Ticket Jira real
jira_client.py llama a la REST API v3, resuelve repository_origen desde las labels del ticket.
GET /rest/api/3/issue/:key
02
Grafo Neo4j
cypher-shell real: qué depende del componente afectado, en qué lenguaje.
query MATCH ()<-[:DEPENDS_ON]-()
03
Hallazgos Sonar reales
sonar_client.py consulta /api/issues/search del proyecto real, incluye el hardcoded-credential de AuthService.java.
GET /api/issues/search
03b
Specs de Figma (opcional)
Solo si el ticket trae un link real de Figma: figma_client.py consulta la REST API real (FIGMA_API_TOKEN) del nodo indicado, sin necesitar Figma Desktop abierto. Sin link o sin token, se omite.
GET api.figma.com/v1/files/:key/nodes
04
AI Firewall
Prompt compuesto (jira+grafo+sonar+figma) evaluado; ingress + egress deciden APPROVED/REJECTED. Auth opcional: header X-Firewall-Key (FIREWALL_API_KEY), 401 sin él.
POST :8080/evaluate
05a
Coding agent real (si GITHUB_REPO)
Crea un Issue con el contexto pre-computado y lo asigna al GitHub Copilot coding agent, que corre en la nube con su propio loop de razonamiento y abre un PR. No tiene acceso a los MCP locales.
POST gh issue create + assignee
05b1
Agente local real: coding_agent.py
Sin GITHUB_REPO, con backend (Anthropic/Ollama) disponible: agente real con loop de tool-calling (agent_loop.py, compartido con el juez) — lee/escribe/lista/busca en el repo y consulta los MCP del juez. Confirmación humana antes de cada escritura/comando; si aplica, commitea en una rama nueva.
out copilot_contribution.jsonl · logs/coding_agent_runs.jsonl
05b2
Fallback: gh copilot suggest
Sin GITHUB_REPO ni backend de modelo disponible: sugerencia de un solo tiro, no un agente. Confirmación humana antes de ejecutar; si aplica, commitea en una rama nueva del repo real detectado (donde el usuario está parado), nunca en la rama base original.
out copilot_contribution.jsonl · POST /issue/:key/comment
06
Testing agent (gate real)
Solo caminos 05b1/05b2 con diff aplicado: auto-detecta el stack por archivo de proyecto (pom.xml/*.csproj/go.mod/Gemfile/Cargo.toml/package.json) y corre el test suite real en contenedor descartable — cubre .NET, NestJS, Angular, Ionic, Expo, React Native, Go, Ruby, Rust, Java, Python sin tocar el script. Determinístico, no un LLM. Si falla: bloquea ahí mismo, el juez ni se llama.
run run_module_tests.sh (auto-detección)
07
Agente juez (Claude/Ollama + MCP real)
Corre en tu máquina (a diferencia del coding agent en la nube), así que se conecta de verdad a mcp-neo4j-cypher y mcp-server-qdrant para VERIFICAR el impacto y el histórico, no solo confiar en el texto. Ve tambien el resultado del testing agent. Audita tanto APPROVED como REJECTED, pero su poder es asimétrico: sobre APPROVED puede bloquear (comenta, transiciona, marca la rama); sobre REJECTED solo puede alertar — nunca revierte el rechazo del firewall. Backend Anthropic → Ollama local (ambos alcanzables desde run_poc_loop.sh y orchestration.py) → se omite sin frenar la corrida.
tools neo4j-cypher · qdrant-rag · out judge_verdicts.jsonl

Lógica de decisión del firewall

orden estricto — egress solo corre si ingress pasa

paso 1 Ingress · Jailbreak
texto normalizado contiene rm -rf, drop table, ignore previous instructions…
⇒ 403 REJECTED — gh copilot nunca se invoca; el juez SÍ audita este rechazo, pero solo para alertar si fue un falso positivo — nunca lo revierte
paso 2 Egress · Fuga de datos
regex sobre el prompt real: password=, secret_key=, patrón Azure key
⇒ redactions_applied ≥ 1 reemplazo por [REDACTED_CORPORATE_SECRET]
paso 3 Veredicto final
200 APPROVED sanitized_prompt no nulo → gh copilot explain real

  1. ＋Jira real — REST API v3, no JSON local; el escenario lo decide el ticket, no un flag.
  2. ＋SonarQube real — escanea sample-repo/ de verdad; la credencial hardcodeada en AuthService.java es un hallazgo real, no fabricado.
  3. ＋Grafo poblado desde Azure DevOps — vía MCP, no un seed estático (que queda solo como fallback).
  4. ＋RAG semántico — Qdrant + embeddings locales sobre código e histórico real de Jira/Sonar.
  5. ＋Cache con TTL — transparente sobre qué se sirvió en vivo vs. cacheado.
  6. ＋check_prereqs.sh — valida los servicios y credenciales antes de arrancar.
  7. ＋Ejecución real, no solo explicación — gh copilot suggest corre de verdad (con confirmación humana) y aplica el cambio en una rama de revisión, nunca en la rama base original.
  8. ＋Repo objetivo: el real, no uno de ejemplo — el pipeline opera sobre el repositorio git donde el usuario está parado al invocarlo (git rev-parse --show-toplevel), y se niega a arrancar si hay cambios sin commitear. sample-repo/ queda solo como referencia opcional.
  9. ＋Métrica de colaboración — copilot_contribution.jsonl + report_sprint_metrics.py miden cuánto sugirió y aplicó Copilot, no solo si el pipeline corrió.
  10. ＋Auditoría en el ticket mismo — jira_client.py comment publica el veredicto del firewall y qué hizo Copilot como comentario real en Jira, visible sin acceso a los logs locales.
  11. ⚠"Pipeline" ≠ CI/CD — run_poc_loop.sh es orquestación manual a demanda. Las ramas copilot/* que crea el fallback local no disparan ningún build/test/re-análisis automático; nada las está observando todavía.
  12. ⚠"Pipeline" ≠ agente — las etapas 1-4 son orquestación determinística, sin razonamiento propio. Solo la etapa 5a (GitHub Copilot coding agent) es un agente real; el fallback 5b es una sugerencia de un solo tiro, sin loop ni acceso a los MCP.
  13. ＋Agente juez con MCP real y poder de bloqueo — Claude (modelo distinto al que actuó) corre local y usa mcp-neo4j-cypher/mcp-server-qdrant de verdad para verificar el contexto, no solo confiar en el texto; si FLAGGED, transiciona el ticket a bloqueado y marca la rama/issue, no solo comenta.
  14. ＋Testing agent determinístico — mvn/npm/pytest reales en contenedor descartable sobre el repo real detectado, gate duro antes del juez (sample-repo/ trae JUnit, Vitest y pytest como ejemplo de referencia).
  15. ＋Testing de UI real — Playwright renderiza una pantalla de login real (public/index.html) en un navegador headless, no solo mockea la lógica en memoria como los tests unitarios.
  16. ＋Monitoreo a nivel de sistema (Falco) — eBPF en tiempo real sobre los contenedores poc-*, complementario al firewall/juez semántico. Requiere privileged: true; en Windows/Docker Desktop depende del soporte de eBPF de la VM WSL2. scripts/check_falco_alerts.py correlaciona las alertas con la ventana de cada corrida (comentario en Jira + webhook opcional), en vez de dejarlas solo en el archivo.
  17. ＋Figma MCP real (interactiva) — Copilot trae medidas/colores/tokens del archivo abierto en Figma Desktop al construir UI, en vez de inventar valores. Un humano decide qué frame.
  18. ＋Figma REST API real (automatizada) — figma_client.py: si el ticket trae un link de Figma y hay FIGMA_API_TOKEN, el pipeline suma specs reales del nodo indicado al prompt compuesto, sin necesitar Figma Desktop abierto ni intervención humana.
  19. ＋Evals de precisión — matriz de confusión del juez contra casos etiquetados a mano (evals/run_judge_evals.py); precisión-proxy del coding agent = % de cambios aplicados que pasan el test suite real.
  20. ＋Orquestación real (Prefect) — orchestration.py reemplaza el bash lineal por un flujo con reintentos por paso, estado persistido y UI en :4200, sobre los mismos building blocks.
  21. ＋Testing agent multi-stack — auto-detección por archivo de proyecto en vez de una lista fija de componentes; agregar un backend en .NET/NestJS/Angular/Ionic/Expo/React Native/Go/Ruby/Rust no requiere tocar el script.
  22. ＋JIRA_KNOWN_COMPONENTS configurable — el set de componentes reconocidos por jira_client.py ya no está hardcodeado a los 3 módulos de ejemplo.
  23. ＋Tests unitarios de la herramienta — tests/ (pytest) cubre firewall_proxy.py, jira_client.py, sonar_client.py, cache_utils.py y judge_agent.py con lógica pura/determinista, sin pegarle a servicios reales.
  24. ＋Gate de Docker alineado entre bash y Prefect — orchestration.py omite el testing agent con gracia si Docker no está en el host, igual que run_poc_loop.sh, en vez de fallar la corrida entera.
  25. ＋repository_origen resuelto de forma inteligente — prioriza el campo nativo Components de Jira (fallback a labels), y el set de componentes conocidos se deriva de los nombres de nodo reales en Neo4j en cada corrida, no de una lista estática en .env.
  26. ＋Mejora continua del juez (curación humana) — review_judge_verdicts.py + promote_reviews_to_evals.py convierten corridas reales confirmadas/corregidas por un humano en casos nuevos de judge_eval_cases.jsonl. Sin reentrenar ningún modelo — es curación de dataset.
  27. ＋Smoke test end-to-end real — scripts/smoke_test.sh crea un ticket Jira real descartable + repo git temporal, corre run_poc_loop.sh de verdad y valida las etapas 1-4 sin mocks. La etapa 5 queda afuera a propósito: gh copilot suggest es una TUI interactiva no automatizable.
  28. ＋Modo épica (--epic) — trae una épica + todos sus hijos y arma un prompt combinado, en vez de procesarlos uno por uno. Solo avanza si todos los componentes resuelven al mismo repo_url en el grafo (nueva propiedad por nodo); si no, se niega a trabajar en vez de adivinar.
  29. ＋Agente de código local real (Camino B1) — coding_agent.py: mismo backend dual Anthropic/Ollama y loop de tool-calling que el juez (agent_loop.py, compartido). Lee/escribe/lista/busca en el repo real y consulta los MCP del juez, con confirmación humana antes de cada escritura o comando. Sin backend disponible, cae a gh copilot suggest (B2) como antes.
