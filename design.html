Pipeline de run_poc_loop.sh

sin argumentos — el escenario lo decide el ticket real de Jira. Orquestación a demanda, no CI/CD: nada dispara esto automáticamente cuando aparece un commit nuevo — ver nota en "Estructura del repositorio" más abajo.

01
Ticket Jira real
jira_client.py llama a la REST API v3, resuelve repository_origen desde las labels del ticket.
GET /rest/api/3/issue/:key
02
Grafo Neo4j
cypher-shell real: qué depende del componente afectado, en qué lenguaje.
query MATCH ()<-[:DEPENDS_ON]-()
03
Hallazgos Sonar reales
sonar_client.py consulta /api/issues/search del proyecto real, incluye el hardcoded-credential de AuthService.java.
GET /api/issues/search
04
AI Firewall
Prompt compuesto (jira+grafo+sonar) evaluado; ingress + egress deciden APPROVED/REJECTED.
POST :8080/evaluate
05a
Coding agent real (si GITHUB_REPO)
Crea un Issue con el contexto pre-computado y lo asigna al GitHub Copilot coding agent, que corre en la nube con su propio loop de razonamiento y abre un PR. No tiene acceso a los MCP locales.
POST gh issue create + assignee
05b
Fallback: gh copilot suggest
Sin GITHUB_REPO: sugerencia de un solo tiro, no un agente. Confirmación humana antes de ejecutar; si aplica, commitea en una rama de sample-repo/, nunca en main.
out copilot_contribution.jsonl · POST /issue/:key/comment
06
Testing agent (gate real)
Solo camino 05b con diff aplicado: corre el test suite real del módulo en un contenedor descartable. Determinístico, no un LLM. Frontend corre Vitest (lógica) + Playwright (UI real: renderiza public/index.html en navegador headless). Si falla: bloquea ahí mismo, el juez ni se llama.
run run_module_tests.sh
07
Agente juez (Claude + MCP real)
Corre en tu máquina (a diferencia del coding agent en la nube), así que se conecta de verdad a mcp-neo4j-cypher y mcp-server-qdrant para VERIFICAR el impacto y el histórico, no solo confiar en el texto. Ve tambien el resultado del testing agent. Si FLAGGED: comenta fuerte, transiciona el ticket a bloqueado, marca la rama con un commit BLOCKED BY JUDGE, y retira la asignación del coding agent si corresponde.
tools neo4j-cypher · qdrant-rag · out judge_verdicts.jsonl

Lógica de decisión del firewall

orden estricto — egress solo corre si ingress pasa

paso 1 Ingress · Jailbreak
texto normalizado contiene rm -rf, drop table, ignore previous instructions…
⇒ 403 REJECTED — corta el pipeline, gh copilot nunca se invoca
paso 2 Egress · Fuga de datos
regex sobre el prompt real: password=, secret_key=, patrón Azure key
⇒ redactions_applied ≥ 1 reemplazo por [REDACTED_CORPORATE_SECRET]
paso 3 Veredicto final
200 APPROVED sanitized_prompt no nulo → gh copilot explain real

  1. ＋Jira real — REST API v3, no JSON local; el escenario lo decide el ticket, no un flag.
  2. ＋SonarQube real — escanea sample-repo/ de verdad; la credencial hardcodeada en AuthService.java es un hallazgo real, no fabricado.
  3. ＋Grafo poblado desde Azure DevOps — vía MCP, no un seed estático (que queda solo como fallback).
  4. ＋RAG semántico — Qdrant + embeddings locales sobre código e histórico real de Jira/Sonar.
  5. ＋Cache con TTL — transparente sobre qué se sirvió en vivo vs. cacheado.
  6. ＋check_prereqs.sh — valida los 8 servicios y 3 credenciales antes de arrancar.
  7. ＋Ejecución real, no solo explicación — gh copilot suggest corre de verdad (con confirmación humana) y aplica el cambio en una rama de revisión, nunca en main.
  8. ＋Métrica de colaboración — copilot_contribution.jsonl + report_sprint_metrics.py miden cuánto sugirió y aplicó Copilot, no solo si el pipeline corrió.
  9. ＋Auditoría en el ticket mismo — jira_client.py comment publica el veredicto del firewall y qué hizo Copilot como comentario real en Jira, visible sin acceso a los logs locales.
  10. ⚠"Pipeline" ≠ CI/CD — run_poc_loop.sh es orquestación manual a demanda. Las ramas copilot/* que crea el fallback local no disparan ningún build/test/re-análisis automático; nada las está observando todavía.
  11. ⚠"Pipeline" ≠ agente — las etapas 1-4 son orquestación determinística, sin razonamiento propio. Solo la etapa 5a (GitHub Copilot coding agent) es un agente real; el fallback 5b es una sugerencia de un solo tiro, sin loop ni acceso a los MCP.
  12. ＋Agente juez con MCP real y poder de bloqueo — Claude (modelo distinto al que actuó) corre local y usa mcp-neo4j-cypher/mcp-server-qdrant de verdad para verificar el contexto, no solo confiar en el texto; si FLAGGED, transiciona el ticket a bloqueado y marca la rama/issue, no solo comenta.
  13. ＋Testing agent determinístico — mvn/npm/pytest reales en contenedor descartable, gate duro antes del juez (JUnit, Vitest y pytest agregados a los 3 módulos de sample-repo/).
  14. ＋Testing de UI real — Playwright renderiza una pantalla de login real (public/index.html) en un navegador headless, no solo mockea la lógica en memoria como los tests unitarios.
  15. ＋Monitoreo a nivel de sistema (Falco) — eBPF en tiempo real sobre los contenedores poc-*, complementario al firewall/juez semántico. Requiere privileged: true; en Windows/Docker Desktop depende del soporte de eBPF de la VM WSL2.
